# v0.1.0 Release Checklist

**Status:** Current
**Last updated:** 2026-06-16 08:40 EDT

The gate list for chatter's first public release. Decisions behind these
gates: see the "Decisions, 2026-06-12" section of
[distribution-and-signing.md](distribution-and-signing.md). Check items
off as they land; a successor picking this up cold should be able to
resume from the first unchecked box.

Release shape: one coordinated v0.1.0 GitHub Release containing CLI
binaries (all five cargo-dist targets) and chatter-desktop installers.
macOS: .dmg signed + notarized, CLI binaries codesigned. Windows:
unsigned with documented SmartScreen workarounds. crates.io publication
is deferred.

## Phase 0: Decisions and baseline

- [x] v0.1.0 release shape decided and recorded (2026-06-12; see
      distribution-and-signing.md "Decisions").
- [x] macOS CLI signing decision: codesign via cargo-dist

## Track A: CLI command surface review (merge/pipeline family)

Scope rule check for the transcript-merge command family (adjudicate,
batch, merge-preflight, pipeline, sanity-scan, speaker-id,
transcript-merge): each subcommand must be useful to CHAT users in
general, per the repo scope rule in CLAUDE.md.

- [ ] Classification memo: each subcommand (CLI layer AND its
      talkbank-transform backing module) dispositioned as
      "stays, general-purpose" or "moves downstream"; maintainer
      sign-off recorded.
- [ ] Any moving subcommand: downstream replacement verified
      (byte-identical replay on a real session) BEFORE removal here;
      removal covers cli/args, dispatch, book pages, SUMMARY.md.
- [ ] Every staying subcommand: accurate `--help`, a book page matching
      actual behavior, documented exit-code contract, at least one
      subprocess-level integration test.
- [ ] README flagship feature list matches the post-review surface.

## Track B: Signing and release CI

- [ ] Local macOS desktop signing: run the scripted ephemeral-keychain
      build (sign + notarize + staple). The signing identity is supplied
      via the `APPLE_SIGNING_IDENTITY` environment variable at build
      time, never hardcoded in tauri.conf.json; no entitlements file is
      needed for a Developer ID app with hardened runtime. Note: macOS
      securityd only authorizes Developer ID key use from an interactive
      (Aqua) login session; run the script from a desktop terminal, not
      over SSH or inside tmux. The locally built .dmg must pass
      `codesign -dv --verify` and
      `spctl -a -t open --context context:primary-signature -v`, and
      Gatekeeper must open it with no override on a second account.
- [ ] `.github/workflows/release-desktop.yml` (tauri-action): macOS
      (signed + notarized), Windows (unsigned NSIS or MSI, pick one),
      Linux (AppImage/deb). Secret NAMES documented in
      `book/src/contributing/ci-and-release.md`.
- [ ] cargo-dist `macos-sign = true` wired in dist-workspace.toml with
      the CODESIGN_* secrets.
- [ ] Windows unsigned UX: SmartScreen walkthrough with screenshots in
      the install docs; verified by a Windows tester using only the
      docs.
- [ ] Signing succession documented: who holds the cert, where CI
      secrets live, what a successor rotates (distribution-and-signing
      open question closed).

## Track C: Audit and documentation

Review methodology: reviews emit issue lists, not edits. Severity
ladder: P0 fix now (user-reachable panics, wrong output on documented
surfaces, license problems, leaked non-public data); P1 fix if cheap,
else document as a known limitation; P2 to `audit-backlog.md` (becomes
GitHub issues after release). Mechanical commits (doc comments,
headers) stay separate from behavioral commits (TDD required).

- [ ] Pass 0 tooling sweep green or triaged: fmt, two-pass clippy,
      cargo doc, cargo audit, cargo deny (license compatibility),
      cargo machete, doc-tests; spec/tools workspace; desktop build.
- [ ] Pass 1 doc comments: `#![warn(missing_docs)]` at zero on the
      first-wave crates (tree-sitter-talkbank, derive, model, cache,
      parser, re2c, transform); every pub item accurate on held-back
      crates (cli, lsp, send2clan).
- [ ] Pass 2 per-crate correctness/efficiency review, bottom-up; dated
      findings table per crate; zero open P0s.
- [ ] Pass 3 test quality: llvm-cov baseline archived; every shipping
      CLI subcommand has a subprocess-level integration test;
      parser-equivalence and reference-corpus gates green.
- [ ] Pass 4 docs accuracy: every user-guide page's commands literally
      run against the built binary; README rewritten as the public landing
      page (install one-liners, experimental designations, release
      badge, talkbank-llm in the crate table).
- [ ] Pass 5 CLAUDE.md accuracy: all sub-project CLAUDE.md files
      spot-verified by running their documented commands.
- [ ] Desktop app experimental marking present in README and install
      docs.
- [ ] Fixture/corpus provenance manifest: every tracked .cha under
      corpus/reference/, test fixtures, and spec examples confirmed
      synthetic, openly licensed, or sanitized; maintainer sign-off.

## Release day

- [ ] Repo settings verified: Actions enabled, secrets present
      (CODESIGN_* and APPLE_* lists from Track B, plus the updater
      `TAURI_SIGNING_PRIVATE_KEY` / `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`),
      branch protection re-applied per the book page.
- [ ] **Auto-update wired (see distribution-and-signing.md "Auto-update"):**
      the Tauri updater keypair is generated; `tauri.conf.json`
      `plugins.updater.pubkey` holds the REAL public key, not the
      `REPLACE_WITH_TAURI_UPDATER_PUBLIC_KEY` placeholder (hard blocker:
      `tauri build` also needs `TAURI_SIGNING_PRIVATE_KEY`, so a keyless
      desktop build fails); the CLI `install-updater` is on
      (`dist plan` lists the `-update` artifacts).
- [ ] Tag v0.1.0 pushed; release.yml and release-desktop.yml both
      complete; release contains CLI archives + checksums + installer
      scripts + .dmg + Windows installer + Linux bundles + the desktop
      updater bundles and `latest.json`; body links CHANGELOG.md.
- [ ] Download smoke tests as an unauthenticated user: macOS install
      one-liner + .dmg open with no Gatekeeper override; Windows
      one-liner + installer via the documented SmartScreen steps;
      Linux musl tarball in a bare container; `latest.json` resolves at
      the release and is signature-valid. Results recorded here.
- [ ] talkbank.org software page live, linking the GitHub Release.
- [ ] talkbank.org-hosted JSON Schemas published so the canonical `$id`
      URLs the book promises resolve. One GENERATED schema exists and just needs
      publishing: `talkbank.org/schemas/v0.1/chat-file.json` (built from the
      Rust source of truth, `schema/*.schema.json` via the `generate_schema`
      test; cited in `json-schema.md` and `quick-start.md` where consumers are
      told to "follow the v0.1 URL"); release day copies that artifact to the web
      host. NOT a publish requirement for v0.1.0: `.../merge-overrides.json` is
      reserved but its schema is NOT yet generated (`OverrideFile` does not yet
      derive `schemars::JsonSchema`); `merge-overrides.md` now documents that URL
      as "reserved but not yet published", so its 404 is a documented-pending
      state, not a broken promise. Re-run the external link check (an external-URL
      lychee pass over the book, excluding the private `github.com/TalkBank`
      self-links) and confirm the only remaining non-200s are documented-pending
      (`merge-overrides.json`) or identifier-only. Note: the XML namespace URI
      `http://www.talkbank.org/ns/talkbank` (cli-reference.md) is an identifier,
      not a hosted page, and is expected not to resolve; do not "fix" it.
- [ ] talkbank.org error-code reference published at `talkbank.org/errors/<CODE>`.
      The BINARY itself emits these URLs in validation output
      (`ErrorCode::documentation_url` -> `https://talkbank.org/errors/E305`), and
      the LSP makes each error code a clickable link to them
      (`codeDescription`), so they are user-visible, not just doc links. Until
      published they 404. Source of truth for the content is the in-book
      `chatter/user-guide/validation-errors.md` plus `spec/errors/`; publishing
      can be a static per-code page or a redirect to the in-book reference.
      Verify a sample (`E305`, `E316`, `E705`) resolves post-publish.
- [ ] Development unfrozen; post-release protocol in
      `book/src/contributing/ci-and-release.md` is the ongoing process.
